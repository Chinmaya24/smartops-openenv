🤖 AI Agents

Three specialized agents collaborate via shared memory to process each email.

🧠
Triage Agent
Reads email → classifies category + assigns urgency 1–5 using keyword matching
💬
Response Agent
Reads shared memory → generates professional reply based on category + urgency
🚨
Escalation Agent
Reads urgency → decides escalation flag and priority level 1–5

Agent Decision Logic

# Triage Agent — keyword-based classification
if "outage" or "system down" or "production":
    → category: technical, urgency: 5

if "refund" or "charged" or "billing":
    → category: billing, urgency: 2

if "login" or "password" or "access":
    → category: technical, urgency: 3

default:
    → category: general, urgency: 1

# Response Agent — template-based replies
if billing:
    → "We are processing your refund..."

if technical + urgency >= 4:
    → "We are aware of the issue and fixing it urgently."

if technical:
    → "We will help you resolve this issue."

default:
    → "Thank you for contacting support..."

# Escalation Agent — urgency-based rules
if urgency >= 4: → escalated: True,  priority: 5
if urgency >= 3: → escalated: False, priority: 3
default:         → escalated: False, priority: 2

Shared Memory Flow

shared_memory = {}

# After Triage:
shared_memory = {"category": "billing", "urgency": 2}

# After Response:
shared_memory = {"category": "billing", "urgency": 2,
                 "response": "We are processing your refund..."}

# After Escalation:
shared_memory = {"category": "billing", "urgency": 2,
                 "response": "We are processing your refund...",
                 "escalated": False, "priority": 2}

🎯 Reward Function

The reward function evaluates agent performance across 4 weighted components.

Reward Weights

Category Match
0.4
Response Keywords
0.3
Escalation Correct
0.2
Priority Correct
0.1

Formula

score = category_match      × 0.4
      + response_keywords   × 0.3
      + escalation_correct  × 0.2
      + priority_correct    × 0.1
      - inefficiency_penalty        # if step_count > 4

# Clipped to [0.0, 1.0]
# Partial progress signals at each step

Why These Weights?

FactorWeightReason
Category0.4Wrong category = wrong department = customer frustration
Response0.3Reply quality directly impacts customer satisfaction
Escalation0.2Wrong escalation wastes senior team resources
Priority0.1Fine-tuning — less critical than the others

🐳 Deployment

Dockerfile

FROM python:3.10-slim

RUN useradd -m -u 1000 user
USER user
ENV PATH="/home/user/.local/bin:$PATH"

WORKDIR /app
COPY --chown=user ./requirements.txt requirements.txt
RUN pip install --no-cache-dir --upgrade -r requirements.txt
COPY --chown=user . /app

EXPOSE 7860
CMD ["python", "-m", "uvicorn", "server.app:app", \
     "--host", "0.0.0.0", "--port", "7860"]

Run Locally

# Install dependencies
pip install -r requirements.txt

# Start server
python -m uvicorn server.app:app --host 0.0.0.0 --port 7860

# Test
curl -X POST http://localhost:7860/reset

💬 Interview Prep

Explain your project in 30 seconds ▼
"I built an AI system that automatically handles customer support emails. When an email arrives, three AI agents work together — one classifies the email, one writes the reply, and one decides if it needs escalation. The whole system is built as a reinforcement learning environment following Meta's OpenEnv standard, deployed on Hugging Face Spaces, and connected to Gmail via n8n for full automation with zero human interaction."
What is OpenEnv? ▼
"OpenEnv is Meta's open-source framework for building standardized reinforcement learning environments. Think of it like OpenAI Gym but for real-world tasks and LLM post-training. It defines a standard API — reset(), step(), state() — so any RL agent can interact with any environment in the same way. I implemented all 3 methods in my FastAPI server with full spec_version 1 compliance."
Why multi-agent instead of single agent? ▼
"Real support teams have specialists — a billing specialist handles refunds differently than a technical specialist handles outages. By splitting into specialized agents, each one can be optimized for its specific task. They also share memory so decisions are collaborative. This mirrors real enterprise workflows and improves accuracy compared to a single generalist agent."
How does your reward function work? ▼
"It has 4 weighted components — category accuracy at 0.4 weight (most important), response quality at 0.3, escalation correctness at 0.2, and priority at 0.1. There's also an inefficiency penalty if the agent takes more than 4 steps. This gives partial rewards at each stage so the RL agent gets learning signal even for partially correct outputs, which speeds up training."
What is PPO and why did you use it? ▼
"PPO stands for Proximal Policy Optimization. It's an RL algorithm that updates the policy in small, safe steps to prevent large destabilizing updates. I chose it because it's stable, sample-efficient, and the industry standard — used by OpenAI and Meta. Compared to vanilla policy gradient, PPO is more stable, and compared to TRPO, it's simpler to implement. I used Stable-Baselines3's PPO implementation."
What was your biggest challenge? ▼
"Two main challenges. First, getting n8n expressions to evaluate correctly — the Gmail Send node was treating {{ $json.From }} as plain text instead of evaluating it because the Fixed/Expression mode toggle wasn't obvious. Second, aligning the openenv.yaml format with the hackathon validator which required spec_version: 1 format — earlier versions used a different schema that failed validation."
How would you scale this system? ▼
"Several improvements: Add Redis for distributed shared memory between agents, PostgreSQL for email logging and analytics, Celery for async processing of high-volume emails, deploy n8n on Railway or Render for 24/7 automation without needing my PC on, upgrade agents from rule-based to LLM-based using Llama or GPT for better response quality, and add a feedback loop where customer replies improve the model over time."
What is Docker and why did you use it? ▼
"Docker packages the entire application — code, dependencies, configuration — into a container that runs identically anywhere. Without Docker, 'it works on my machine' is a common problem. I used it because Hugging Face Spaces requires Docker for custom deployments, and it ensures the app runs the same locally and in production. My Dockerfile uses port 7860 which is required by HF Spaces."
What is FastAPI and why not Flask? ▼
"FastAPI is faster than Flask, has automatic API documentation at /docs via Swagger UI, built-in Pydantic validation for type-safe inputs, and async support for high concurrency. For an RL environment that needs to handle multiple requests efficiently, FastAPI's performance and automatic validation are important advantages over Flask."
